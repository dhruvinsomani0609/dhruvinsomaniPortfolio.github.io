The Problem Space

Managing multiple backend services manually and distributing workloads evenly is complex in distributed systems. This project tackles three core challenges: fairly distributing client requests across servers without a single bottleneck, automatically detecting and failing over around unhealthy servers with zero downtime, and providing real-time visibility into live system state — all using Python's native asyncio rather than heavyweight frameworks.

Architecture / 02

Request Flow — Start to Finish

A three-tier architecture where the load balancer acts as the sole entry point, routing traffic intelligently across a pool of independent backend servers while a parallel health-check loop continuously monitors their availability.

01
📨
Client Request
Client sends a JSON operation to the load balancer on port 12000. Example: {"operation":"prime","value":"17"}.
02
⚙️
Round-Robin Selection
The load balancer picks the next healthy backend via choose_backend_server() — cycling through A, B, C, skipping any marked down by the health checker.
03
📤
Forward & Execute
The request is forwarded to the selected backend (port 13001–13003). The backend runs the operation — Fibonacci, prime checking, string reversal, etc.
04
📥
Log & Respond
The response is logged to Redis (LPUSH → LTRIM to 100 entries), then returned to the client with a server_id field for traceability.
05
❤️
Parallel Health Checks
Three concurrent async coroutines run continuously — one per server. Every 5 seconds each sends a PING, awaits a PONG, and updates the server's status in-memory and in Redis hash backend_health.
06
📊
Dashboard Monitoring
The Flask dashboard reads Redis in real-time — displaying per-server health status, the request count from requests_processed counter, and the live log feed.
Key Decisions / 03

4 Pivotal Engineering Choices

Each architectural decision balanced educational clarity with demonstrating production-grade patterns — making the internals visible without oversimplifying them.

01
asyncio Over Threading
⚠ Problem Thread-per-client concurrency is easy to reason about but expensive — each thread consumes ~8MB of stack memory, and context switching becomes a bottleneck at any meaningful concurrency level.
✓ Solution Built the primary load balancer on Python asyncio: a single-threaded event loop that handles thousands of concurrent connections by suspending at I/O waits and resuming coroutines as data arrives — no thread overhead.
A threading-based version was kept for educational comparison. The async version is demonstrably lighter and serves as a model for modern Python concurrency.
02
Round-Robin with Health Awareness
⚠ Problem A naive round-robin without health awareness routes requests to crashed servers — the client hangs, the load balancer blocks, and the next server in rotation still gets skipped even though it's healthy.
✓ Solution The choose_backend_server() function checks server_status for every candidate during iteration. Failed servers are silently skipped in O(n) time, protected by an asyncio.Lock to prevent pointer race conditions.
The load balancer automatically routes around failures without any configuration change — demonstrating fault tolerance through algorithmic awareness, not infrastructure.
03
PING/PONG Health Check Protocol
⚠ Problem Sending real computational requests as health probes would skew metrics, consume backend resources unnecessarily, and give false positives — a server could handle PING but fail on actual workloads.
✓ Solution Each backend server handles a special {"type":"PING"} message type separately from computation requests, returning {"type":"PONG"} immediately. Three health check coroutines run in parallel, each on a 5-second loop.
Near-zero overhead on backends. Clear pass/fail semantics. Decouples availability detection from operation correctness.
04
Redis for Centralized Observability
⚠ Problem Log files and print statements don't work for a distributed system — they're local to the process, require `tail -f` to observe, and aren't queryable from a web UI without a full log aggregation stack.
✓ Solution All events are pushed to Redis via async LPUSH (with LTRIM capping at 100). Health states live in a Redis HASH. The Flask dashboard reads both with a single request — no file I/O, no polling processes.
Real-time, persistent, queryable observability using one dependency (Redis) already containerized. Demonstrates centralized logging as an architectural pattern.
Core Code / 04

The Async Round-Robin Engine

The two functions at the heart of the system: health-aware server selection and fault-tolerant request forwarding with automatic retry on failure.

load_balancer_async.py · core routing functions
async def choose_backend_server(): global next_server_index async with status_lock: # prevent race on shared pointer for _ in range(len(backend_servers)): server = backend_servers[next_server_index] next_server_index = (next_server_index + 1) % len(backend_servers) host, port, _ = server if server_status.get((host, port), False): return server # first healthy server found return None # all servers are down async def forward_request(request: dict): attempts = 0 while attempts < len(backend_servers): server = await choose_backend_server() if not server: break host, port, identifier = server try: reader, writer = await asyncio.open_connection(host, port) await send_json(writer, request) response = await recv_json(reader) writer.close(); await writer.wait_closed() if response: response['server_id'] = identifier # tag source for traceability await log_to_redis(f"Forwarded to server {identifier}: {response}") return response except Exception as e: async with status_lock: server_status[(host, port)] = False # mark down, retry next await log_to_redis(f"Server {identifier} failed: {e}") attempts += 1 return {"error": "All backend servers are down or unresponsive."}

Why the Lock Matters

The asyncio.Lock around next_server_index is the critical detail. Without it, two concurrent coroutines could both read the same pointer value, both increment it, and both route to the same backend — violating the round-robin guarantee. The lock ensures pointer advancement is atomic: only one coroutine can select and advance the index at a time, while all others suspend and wait. This is asyncio's equivalent of a mutex without the thread overhead.

Challenges / 05

Problems & How They Were Solved

Building a distributed system on a single machine surfaces real distributed systems problems — just at a smaller scale. Each challenge taught a pattern applicable at any scale.

🔄
Race Conditions on Shared State
Two async coroutines handling simultaneous client connections both called choose_backend_server() at the same instant — reading and then both incrementing next_server_index, causing the same server to be selected twice in a row and breaking round-robin fairness.
✓ Wrapped the pointer read-increment-return sequence in an asyncio.Lock. Only one coroutine advances the index at a time; others suspend at the lock boundary and resume with the updated value.
🔌
Silent Connection Failures
When a backend server was stopped mid-session, the next request to that server would hang indefinitely — the OS kept the TCP socket open briefly, so connection succeeded but no response arrived. The client waited forever.
✓ Added a try/except in forward_request() that catches any connection or I/O exception, immediately marks the server as False in server_status, and retries with the next server in the rotation — returning an error only if all are down.
📋
Log State Lost on Restart
When the load balancer process was restarted, the in-memory server_status dict reset to all-healthy. The system needed 5+ seconds to re-discover crashed servers via health checks — during which it happily routed requests to downed backends.
✓ Health status is now written to Redis hash backend_health on every check cycle — providing an external source of truth. On restart, the system could seed its in-memory state from Redis before accepting connections, minimizing the blind window.
Learnings / 06

Three Principles Internalized

The lessons from building production-grade distributed patterns at small scale. Each one transfers directly to real-world distributed systems work.

01
Concurrency Is Not Parallelism
Python's asyncio runs on a single thread — there is no literal parallelism. Yet it handles thousands of concurrent connections by exploiting I/O wait time. The core insight is that most distributed system bottlenecks are I/O-bound, not CPU-bound. Coroutines that yield at every network call give the event loop space to handle other connections — achieving concurrency without the overhead of threads or processes.
02
Design for Failure, Not for Success
The naive approach assumes all servers are up and routes accordingly. Production systems assume servers will fail and build routing logic around that assumption. The retry loop in forward_request() — not the happy path — is where the real engineering lives. Every distributed system feature should be designed with its failure mode as the primary requirement, not an afterthought.
03
Observability Is Architecture
Logging to Redis wasn't a UI nicety — it was a foundational architectural decision. By treating observability as a first-class component (not print statements), the system became debuggable, monitorable, and extensible without touching core routing logic. The Flask dashboard reads from Redis independently of the load balancer. In distributed systems, if you can't observe it, you can't operate it.
