The Problem Space

Enterprise document extraction tools are rigid. When invoice formats change or new schemas are introduced, hardcoded systems fail silently. Organizations needed a scalable, secure platform where users could dynamically define and evolve JSON/CSV/XML extraction schemas — without breaking historical job data or causing billing inconsistencies under concurrent upload load.

Architecture / 02

Event-Driven Process Flow

A highly decoupled, serverless architecture where each component handles a single responsibility — from schema definition at the client all the way to AI-powered structured extraction by worker nodes.

01
📋
Schema Management
Users define JSON/CSV/XML output formats ("Tags") via React frontend with live Monaco Editor for real-time try/catch syntax validation.
02
📤
Document Ingestion
PDFs and images are uploaded directly to AWS S3. High-volume concurrent batch uploads are handled automatically by S3 event triggers.
03
⚡
Lambda Event Trigger
S3 triggers a serverless Lambda that calculates file size (bytes → MB), page count, and determines the dynamic credit cost per document.

Atomic Billing
A PostgreSQL RPC uses SELECT FOR UPDATE row-level locking to safely deduct credits and log an immutable transaction — zero race conditions.

Core Decisions / 03

6 Pivotal Engineering Choices

Each decision resolved a fundamental tension between scalability, data integrity, and developer experience — arrived at through shipping real production issues, not theory.

01
Tag Versioning Engine
⚠ Problem A simple SQL UPDATE to a Tag's schema would silently corrupt all historical jobs that relied on the old format — no audit trail, no rollback path.
✓ Solution Dual-path update logic: light metadata changes (name, description) update in-place; schema changes archive the old tag (is_active = false) and create a new versioned row.
100% backward compatibility for all historical job data. Users freely evolve schemas without breaking previous extraction records.
02
Atomic Credit Billing via DB RPC
⚠ Problem 100 simultaneous S3 uploads spawned 100 parallel Lambda instances. Each read the same stale wallet balance — race conditions caused credit balance to be overwritten, losing revenue.
✓ Solution Moved all credit deduction math from Python into a Supabase RPC using SELECT ... FOR UPDATE row-level locking, making PostgreSQL the single atomic source of truth.
Guaranteed atomic transactions. Concurrent Lambda uploads are safely serialized at the database level — zero balance discrepancies.
03
Strict Job Creation Order (FK Integrity)
⚠ Problem Logging financial transactions before the parent job row was committed caused FK constraint violations — orphan ledger records with no associated job, corrupting the credit dashboard.
✓ Solution Enforced strict insert ordering: Generate UUID → Insert jobs row → Insert credit_transactions row → Insert job_files linked rows → push to SQS.
Zero orphan transactions. The credit dashboard reliably maps every cost entry to its job, tag, and document via LEFT JOINs.
04
Cross-Tenant Security (org_id Scoping)
⚠ Problem Initial backend code lacked org_id validation on Supabase queries. User A could query User B's tag_id directly, exposing a cross-tenant data leakage vulnerability.
✓ Solution All Supabase queries now resolve the org_id from the verified JWT user_id and attach .eq('org_id', org_id) to every query — combined with RLS policies.
Complete tenant data isolation. No user can access another organization's tags, jobs, or billing data — even with a valid JWT.
05
Live Client-Side Schema Validation
⚠ Problem Users submitting malformed JSON through React forms reached the Python Lambda parser and caused unhandled 500 Server Errors that corrupted job state.
✓ Solution Integrated Monaco Editor with real-time try/catch JSON parsing on onChange. The Save button is automatically disabled until all syntax errors are resolved in the editor.
Eliminated backend 500 errors from bad schema payloads. Frontend as the first line of defense; backend SQL constraints as the last.
06
document_id FK for Direct Uploads
⚠ Problem The credit dashboard showed "System Adjustment" for direct document uploads because transactions had no link to individual files — only batch job_ids were tracked.
✓ Solution Altered the credit_transactions table to include a document_id FK. The RPC now uses COALESCE to look up filenames from documents for uploads or job_files for batches.
Zero ambiguity in the ledger. Every credit entry shows the exact file or batch it was charged for — direct uploads and batch jobs unified in one dashboard.
Final Code / 04

The Atomic Billing RPC

The critical pivot from application-layer math to data-layer integrity. This PostgreSQL function replaced fragile Python race-prone calculations and became the financial backbone of the platform.

deduct_document_fee.sql · Supabase RPC
CREATE OR REPLACE FUNCTION deduct_document_fee( p_org_id uuid, p_amount numeric, p_document_id uuid ) RETURNS numeric LANGUAGE plpgsql SECURITY DEFINER AS $$ DECLARE v_wallet_id uuid; v_new_balance numeric; BEGIN -- Step 1: Lock the wallet row — prevents concurrent reads SELECT wallet_id INTO v_wallet_id FROM public.credit_wallets WHERE org_id = p_org_id FOR UPDATE; -- ← The critical row lock -- Step 2: Atomically deduct and capture new balance UPDATE public.credit_wallets SET balance = balance - p_amount, updated_at = now() WHERE wallet_id = v_wallet_id RETURNING balance INTO v_new_balance; -- Step 3: Write immutable ledger entry linked to the specific document INSERT INTO public.credit_transactions ( wallet_id, amount, balance_after, transaction_type, service_type, document_id ) VALUES ( v_wallet_id, -p_amount, v_new_balance, 'DEBIT', 'DOCUMENT_UPLOAD', p_document_id ); RETURN v_new_balance; END; $$;

Why This Matters

Before this RPC, Python code would GET balance → subtract → UPDATE in three separate network round-trips. Between steps 1 and 3, another Lambda could read a stale balance. The RPC collapses all three operations into a single atomic transaction with a row-level lock — making it physically impossible for two Lambdas to corrupt the same wallet simultaneously. The added document_id column also links every deduction to a specific file, making the credit ledger fully auditable.

Diff Payloads

Partial Update Refactor

Refactored the React onSubmit handler to diff current state against original references, sending only modified fields in the PUT request. The Python Lambda was updated to dynamically patch missing fields from existing database records — eliminating unnecessary bandwidth and preventing accidental overwrites of unchanged fields in concurrent environments.

Dynamic Ledger

Intelligent Credit Dashboard

Altered the credit_transactions table to add a document_id foreign key. Rewrote the dashboard RPC with a polymorphic COALESCE subquery: direct document uploads resolve the filename from the documents table, while batch job transactions look it up via job_files. Every credit entry now provides exact auditing clarity.

Atomic Billing

Database-Level Row Locking

Stripped financial arithmetic out of the Python application layer entirely. Shifted credit deductions into a PostgreSQL RPC utilizing SELECT ... FOR UPDATE. This enforces strict row-level database locks, queuing concurrent serverless S3 transactions atomically and completely eliminating financial race conditions.

Immutable Data

Dual-Path Versioning Engine

Engineered a dual-path versioning system for user-defined schemas. Light metadata updates execute in-place, while structural schema changes archive the original row (is_active = false) and generate a newly versioned row. This ensures absolute backward compatibility for historical jobs while allowing frictionless schema evolution.

Bug Fixes / 06

Notable Bugs Squashed

Five production bugs that ranged from silent data corruption to security vulnerabilities — each fixed with targeted, minimal code changes once the root cause was identified.

Bug Root Cause Fix
The Distributed Concurrency Race Condition
The serverless architecture meant 100 simultaneous S3 document uploads spawned 100 parallel AWS Lambdas. The application layer read the wallet balance, subtracted the cost, and wrote it back. Simultaneous Lambdas read the same stale balance, silently overwriting each other and causing massive revenue leakage.
Stripped financial math out of the Python application layer. Moved deductions into a PostgreSQL RPC utilizing SELECT ... FOR UPDATE, enforcing strict row-level database locks to queue transactions atomically.
The Orphaned Ledger Foreign Key Crash
Job dispatch Lambdas failed abruptly during execution. The code attempted to log a credit deduction in the credit_transactions table referencing a job_id that had not yet been committed to the jobs table, triggering a hard SQL Foreign Key constraint violation.
Re-architected the execution sequence to satisfy relational integrity: UUID Generation → jobs INSERT → credit_transactions INSERT → job_files INSERT → AWS SQS Dispatch.
Cross-Tenant Data Leakage
Backend Supabase queries lacked org_id filtering. Any authenticated user could query another organization's tags by passing a known tag_id — a silent multi-tenant isolation failure.
All queries now resolve org_id from the JWT user_id and chain .eq('org_id', org_id) to every Supabase query. RLS policies provide a second layer.
The Ghost Document Ledger Issue
The React credit dashboard displayed "System Adjustment" instead of actual filenames for direct document uploads. The ledger table only tracked batch job_ids, meaning single-file uploads bypassed batch creation and left the SQL LEFT JOIN completely blind.
Evolved the live database schema to include a document_id foreign key. Engineered a polymorphic COALESCE subquery inside the dashboard RPC to dynamically route lookups to either the batch job hierarchy or the direct document table based on transaction type.
The Historical Schema Corruption Anomaly
When users updated an IDP extraction schema (JSON) via a standard PUT request, the in-place SQL UPDATE altered the schema globally. This silently corrupted the historical context of already-processed jobs that relied on the older schema format for review.
Engineered a dual-path versioning engine. Heavy schema changes now archive the original row (is_active = false) and insert a newly versioned row, while light metadata updates execute in-place, ensuring absolute backward compatibility.
Learnings / 07

Four Principles Internalized

Lessons that only emerged from building a production system under real constraints — not from theory, but from hitting the hard edges of each architectural choice.

01
Event-Driven, Decoupled Asynchronicity
production-level document processing using heavy OCR and expensive LLMs (the IDP worker nodes) cannot survive inside a synchronous web request lifecycle. Trying to process thousands of documents simultaneously while keeping a frontend spinner alive is a recipe for connection exhaustion and timeout failure. We refactored to a fully event-driven, serverless pipeline: S3 ingestion triggers billing Lambda, which dispatches job IDs to AWS SQS. Only by strictly decoupling ingestion (sync) from intensive processing (async) did we achieve linear scaling and predictable backend performance.
02
Never Trust the Client
Client-side validation exists for user experience, not security. Because any frontend UI can be trivially bypassed by direct API calls, the backend must act as the ultimate gatekeeper. Independently sanitizing payloads, re-validating data, and enforcing strict database constraints ensures system integrity regardless of how a request originated. Defense in depth is non-negotiable.
03
Stateless vs. Stateful at Scale
AWS Lambdas scale infinitely — but relational databases do not scale infinitely with them. Bridging the two requires moving critical state mutations (wallet balances) into atomic database functions. Application-layer math doesn't survive concurrency; database-layer row locks do.
04
Silent Failures Are the Worst Kind
RLS blocking an UPDATE with no error is harder to debug than a thrown exception. The cross-tenant bug and the RLS silent failure both taught the same lesson: always verify writes have actually committed. Returning affected row counts — not just success codes — is non-negotiable.
