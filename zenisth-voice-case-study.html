The Problem Space

Traditional vector database RAG approaches rely on random text chunking and cosine similarity — destroying document structure and leading to inaccurate retrieval for long, complex documents like financial reports, SOPs, and manuals. Zenisth Voice Agent solves this by preserving the entire document hierarchy as a tree, then guiding an LLM to traverse it like a human expert would — section by section, reasoning at each node — to find the exact context needed before generating an answer. On top of that, users never have to type: the entire experience is voice-first via Deepgram.

Architecture / 02

Two Pipelines — Ingestion & Retrieval

The system operates on two separate data flows. The ingestion pipeline builds a hierarchical tree once. The retrieval pipeline traverses that tree on every query — guided at each level by an LLM — until it reaches the exact document section needed.

01
🎙️
Voice Input
User speaks a query into the browser. Deepgram processes the live audio stream in real-time, returning a high-accuracy text transcript with minimal latency.
02
🌳
Tree Root Scan
The query is sent to the backend. The LLM is shown the top-level document tree (Level 1 headings + their summaries) and asked: "Which branch is relevant to this query?"
03
🔍
Drill-Down Traversal
The system recursively drills into the selected branch — Level 2 → Level 3 → leaf nodes — until the LLM identifies a node whose start_index/end_index bounds contain the answer.
04
📄
Context Extraction
The exact text slice identified by the start_index and end_index of the selected node is pulled from the relational DB — no hallucination-prone similarity ranking.
05
💬
Answer Generation
The LLM receives the exact context slice plus the original query and generates a precise, grounded answer — no hallucination from irrelevant vector chunks.

Key Decisions / 03

4 Pivotal Engineering Choices

Each decision was driven by the core constraint: accurate retrieval from complex, long documents without hallucination — while keeping the interface effortless for professionals.

01
Tree-Structured Vectorless RAG
⚠ Problem Standard vector RAG chunks documents without regard for structure. A 200-token chunk from a financial report may contain a paragraph mid-sentence, with no heading context. Similarity search then ranks irrelevant chunks highly based on keyword overlap, not meaning.
✓ Solution Implemented a hierarchical "PageIndex" inspired approach: documents are parsed into a tree of heading nodes with start_index and end_index bounds. An LLM reasons over node summaries level by level to drill to the correct section — no embeddings, no vector DB.
True relevance over mathematical similarity. Works on long, structured documents where vector RAG fails. Exact text bounds eliminate hallucination-prone context windows.
02
Deepgram for Voice-First UX
⚠ Problem Domain professionals (doctors, analysts, legal experts) querying dense documents shouldn't need to type precise technical queries. A voice interface removes the interaction barrier — but browser-native STT has poor accuracy on domain-specific terminology.
✓ Solution Integrated Deepgram for high-accuracy, real-time speech-to-text with live browser audio streaming. The POC was debugged end-to-end — including fixing audio buffer handling that was silently producing null transcripts.
Hands-free, natural interaction. Deepgram accuracy outperforms browser STT on technical vocabulary. Zero-latency feel via live streaming (not batch upload).
03
Docling Over MarkItDown
⚠ Problem MarkItDown was the initial ingestion library but could not reliably distinguish heading levels (Level 1 vs. Level 2 vs. Level 3) in PDFs and DOCX files — collapsing the hierarchy the tree-structured RAG system depended on entirely.
✓ Solution Swapped ingestion library to Docling: a document parsing tool designed explicitly to preserve structural semantics, heading hierarchy, tables, and reading order — producing the heading level metadata the tree builder requires.
Unblocked the hierarchical tree building pipeline. Heading levels now parse accurately, enabling the LLM to traverse meaningful node boundaries during retrieval.
04
Backend Refactor — Monolith to Package
⚠ Problem The original voice agent was a single monolithic Python script. Model names, API keys, and defaults were hardcoded throughout — making it impossible to switch models, deploy to different environments, or update config without touching core logic.
✓ Solution Refactored the entire backend into a structured Python package with strict separation of concerns. All configuration centralized in config.yaml. New REST endpoints (/api/models, /api/config) let the frontend pull settings dynamically — no hardcoding.
DRY, maintainable codebase. Model switching is a config change, not a code change. Frontend and backend are fully decoupled — any client can consume the API.
Core Code / 04

The Tree Traversal Loop

The heart of the vectorless RAG system: an LLM-guided iterative drill-down that reads node summaries at each tree level and selects the branch most relevant to the query — until it reaches a leaf node.

rag/retriever.py · traverse_tree()
def traverse_tree(query: str, tree: dict, db) -> str: current_nodes = tree['children'] # start at root's children (Level 1 headings) while current_nodes: # Build a summary map for the LLM to reason over summaries = { n['id']: f"{n['heading']}: {n['summary']}" for n in current_nodes } # Ask the LLM which node is most relevant to the query selected_id = llm_select_node(query, summaries) if selected_id is None: break # LLM found no relevant branch selected_node = next( (n for n in current_nodes if n['id'] == selected_id), None ) if not selected_node['children']: # Leaf node reached — extract exact text slice return db.get_text_slice( start=selected_node['start_index'], end=selected_node['end_index'], ) # Not a leaf — drill into children at next level current_nodes = selected_node['children'] return None # No relevant section found

Why This Beats Cosine Similarity

A vector similarity search finds chunks whose embedding is close to the query embedding in high-dimensional space. That's a useful heuristic — but it's blind to document structure, prone to topical overlap between unrelated sections, and completely fails when the answer is in a section whose heading doesn't appear in the query. The tree traversal approach lets the LLM reason: "Section 4.3 — Credit Risk Models — is where an answer about loan default probabilities would live" — the same reasoning a human expert would apply. The start_index/end_index system then pulls the exact text — no chunking artifacts, no context bleed.

Challenges / 05

Problems & How They Were Solved

Each blocker here was a lesson in why the choice of underlying tools matters as much as the architecture above them.

🌲
Hierarchy Loss During Ingestion
The initial ingestion library MarkItDown could not reliably distinguish heading levels (L1 vs. L2 vs. L3) in PDFs and DOCX files — collapsing the entire document into a flat Markdown string with incorrect or missing heading depth metadata. This directly blocked the tree builder from constructing a meaningful node hierarchy.
✓ Replaced with Docling — a document intelligence library built explicitly to preserve structural semantics, heading hierarchy, and reading order. Tree construction now receives accurate Level 1/2/3 metadata for every heading node.
🎤
Null Deepgram Transcripts
The Deepgram STT proof-of-concept was returning null transcripts in the browser. The audio was being captured, the WebSocket connection was live — but the final transcript handler received empty data. The bug was silent: no errors, no warnings, just empty results that looked like API failure.
✓ The issue was in audio buffer encoding and chunking before sending to the Deepgram WebSocket. Fixed the browser audio capture pipeline to correctly encode PCM chunks and handle the transcript event sequence, ensuring the final transcript resolves to populated text.
📁
DOCX Format Errors & UI State Bugs
Uploading Microsoft Word documents initially threw format errors — the backend had no DOCX-aware parser. Separately, the Knowledge Document Library had two persistent UI bugs: documents would reappear after refresh despite being deactivated, and activating a document triggered a json not defined runtime error in the frontend.
✓ Added mammoth and docx2txt as DOCX fallback parsers with a format-detection routing layer. Fixed the Knowledge Library state management: corrected the activation handler's JSON serialization call and cleared the leftover default document initialization logic that persisted across page refreshes.
Learnings / 06

Three Principles Internalized

The patterns learned building a production-grade AI document system — each one applicable well beyond this project.

01
Structure Is the Core Asset
The entire vectorless RAG strategy only works because document structure is preserved. The moment you chunk a document randomly — discarding headings, splitting across sections, ignoring hierarchy — you've destroyed the one artifact that makes accurate retrieval possible. Choosing the right ingestion tool (Docling over MarkItDown) wasn't a secondary concern: it was the foundational enabling condition for everything else.
02
Silent Failures Are the Worst Kind
The Deepgram null transcript bug produced no errors — just empty results. The system appeared to work end-to-end but was silently failing at the audio encoding step. In any pipeline that processes external input (voice, files, API) through multiple stages, each stage must emit observable, verifiable signals — not just assume its output is correct. Silent failures cost far more debugging time than loud ones.
03
Config as Code Is Non-Negotiable
A hardcoded model name buried inside a 400-line Python script is a deployment blocker. Centralizing configuration in config.yaml and exposing it via REST endpoints transformed the system from "works on my machine" to something redeployable, testable, and actually configurable. This is the architectural lesson that makes DRY code a product requirement, not just a style preference.
