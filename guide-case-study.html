The Problem Space

Translating a product idea into a visual UI prototype is a slow, manual process. Designers and developers spend hours converting textual requirements into visual interfaces. GUIDE automates this with LLMs — turning a single natural language sentence into a working, styled HTML layout ready for browser preview or direct injection into a Figma canvas.

Architecture / 02

Prompt to Pixels — The Full Pipeline

A simple but effective event chain: user describes a UI in plain English, the LLM reasons about structure and generates HTML + CSS, then the frontend routes the output to either a live browser preview or Figma component nodes.

01
✍️
Prompt Input
User enters a natural language UI description. Example prompts are provided as chip buttons. Multilingual input is supported.
02
⚙️
Backend Processing
The Python Flask backend receives the prompt and wraps it in a structured system prompt that instructs the LLM to output valid HTML + CSS only.
03
🦙
Ollama LLM Inference
The structured prompt is sent to the local Ollama server via /api/chat. LLaMA 3.2 or CodeLlama generates the full UI layout as HTML + CSS.
04
🔄
HTML Returned
The backend returns the raw generated HTML to the frontend. The /preview endpoint also allows direct rendering requests.
05
🖥️
Dual Rendering
Browser: HTML injected into an iframe or opened via blob URL in a new tab. Figma: DOM is parsed — <input>, <button>, <h1> mapped to Figma nodes and injected into the canvas.

Key Decisions / 03

4 Pivotal Engineering Choices

Each decision directly improved generation quality, rendering flexibility, or developer experience — informed by the research paper "GUIDE: LLM-Driven GUI Generation Decomposition for Automated Prototyping."

01
Switched from JSON to HTML-Based LLM Output
⚠ Problem Initial versions instructed the LLM to output structured JSON component trees. JSON format severely limited layout flexibility — nested positioning, inline styles, and responsive designs could not be expressed cleanly.
✓ Solution Switched LLM output target to raw HTML + CSS. HTML is a natural, expressive UI grammar that LLMs understand far better than ad-hoc JSON schemas — and it renders directly in a browser without a translation layer.
Dramatically improved generation quality and layout richness instantly. Removed an entire JSON→DOM translation pipeline from the codebase.
02
Ollama Local LLM Inference
⚠ Problem Cloud LLM APIs (GPT, Gemini) incur per-token costs, have rate limits, and send user prompts to remote servers — unsuitable for a research tool that needed rapid, free iteration and private experimentation.
✓ Solution Integrated Ollama as the local inference runtime. Configuration exposed via OLLAMA_URL and OLLAMA_MODEL env vars, making model switching (llama3.2 ↔ codellama) a one-line config change.
Zero inference costs. Unlimited iteration speed. Complete data privacy. Swappable model backends with no code changes.
03
iframe + Blob URL Preview System
⚠ Problem After obtaining generated HTML, there was no safe, sandboxed way to preview it without running a separate server per generation, risking script injection into the main app page.
✓ Solution Built a dual-mode preview system: (1) iframe renders the HTML inline in a sandboxed container; (2) blob URL wraps the HTML in a temporary Blob object and opens it in a new browser tab for a full-page preview.
Safe, instant UI preview with zero server round-trips for rendering. Users can open any generated UI as a full page in one click.
04
HTML → Figma Node Parser
⚠ Problem Figma doesn't accept raw HTML. Designers using the Figma plugin needed generated UIs to appear as properly structured Figma components — not as flat images or embedded iframes.
✓ Solution Implemented generateFromHTML(): a DOM parser that traverses the generated HTML, maps recognized semantic elements (<input>, <button>, <h1>) to Figma primitives, and injects them into the canvas via the Figma Plugin API.
Designers get editable, structured Figma components — not screenshots. The "Use in Figma" button closes the gap between AI generation and real design workflows.

Why Structured Prompting Matters

Without the system message constraining the output format, LLMs return conversational text, markdown code fences, and explanations — none of which can be rendered as HTML. The system prompt acts as a strict output contract: the LLM is instructed to behave as a UI generation machine, not a chatbot. This single design decision is the difference between a working prototype tool and a code snippet generator.

Challenges / 05

Problems & How They Were Solved

The hardest part of building a prompt-to-UI system isn't the LLM integration — it's dealing with the unpredictability of LLM output and bridging two fundamentally different representation systems: HTML and Figma nodes.

🔀
Inconsistent LLM Output Structure
The LLM sometimes returned markdown-fenced code blocks (```html), prose explanations, or mixed text-and-HTML — none of which render directly in an iframe. The frontend had no reliable way to extract just the HTML.
✓ The system prompt was tightened to explicitly forbid all non-HTML output. A post-processing strip was also added to remove common markdown artifacts before injection.
🧩
HTML→Figma Component Mapping
HTML and Figma have fundamentally different component models. Mapping arbitrary, LLM-generated HTML to Figma's node primitives required handling unknown nesting depths, inline styles, and elements for which Figma has no direct equivalent.
✓ Built a focused element whitelist — only semantic UI tags (input, button, label, h1–h3) are mapped. Unknown elements are skipped gracefully, preventing Figma plugin crashes.
🏗️
Complex & Responsive Layouts
Prompt-based generation degrades on complex UI descriptions. Nested component structures, grid/flexbox layouts, and responsive breakpoints are difficult for a local 8B-parameter model to handle reliably in a single generation pass.
✓ Decomposed prompts into simple, specific UI sections rather than trying to generate a full page at once. RICO dataset integration is planned to provide real layout patterns as few-shot examples in the prompt context.
Learnings / 06

Three Principles Internalized

Building a system where an LLM is the primary code generator — not a developer assistant — surfaces fundamentally different engineering challenges than traditional software.

01
Prompts Are Your Primary API
In LLM-driven systems, the system prompt is the contract between your application and the model. A vague prompt produces vague output. Constraining the model to a strict output format — and explicitly forbidding everything else — is the single most effective technique for making LLM output reliable enough to pipe directly into a rendering pipeline.
02
Domain Representations Don't Translate
HTML and Figma's node model are both UI representations, but they are not isomorphic. Every element mapping decision reveals an impedance mismatch: styles, positioning, inheritance, and layout models all differ. Building an LLM-to-design-tool bridge requires explicitly acknowledging these gaps and designing around them, rather than assuming one maps cleanly to the other.
03
Local Inference Enables New Workflows
Running LLMs locally via Ollama removes the constraints that cloud APIs impose: rate limits, per-call costs, and privacy concerns. This shifts the bottleneck from budget to compute — and opens up tight iteration loops that would be economically impractical with remote inference. For research and rapid prototyping, local inference is worth its setup cost.
